# CRYPTOGRAM Roadmap 2026-04-03

## Guiding Target State
n. Status language strictly reflects evidence: `present`, `partial`, `placeholder`, `requires validation`, `validated`, `externally dependent`.
1. Android Double Ratchet path either implements full ratchet semantics or is relabeled as session encryption everywhere UI/docs touch it.
2. Android MLS uses real X448/keygen/sign/verify and is explicitly experimental until interoperability verification finishes.
3. Desktop settings expose only implemented behavior; unfinished controls are hidden, disabled, or clearly marked non-operational.
4. Tests/reporting separate static wiring, build, runtime, crypto, and device validation tiers before any production claim.
5. Build documentation matches repo reality (or clearly states the external dependency that must be added) for both platforms.
6. “Ready/complete/production” claims close only after QA + Build/Infra gates and documentation sign-off.

## Contradictions to Resolve
| Claim source | Reality | Impact |
|---|---|---|
| `FINAL_STATUS.md` and legacy implementation docs | Ongoing audits/README snapshot call out placeholder status and limited validation | Causes trust erosion and blocks prioritization of remaining work |
| Android docs/status and UI text claiming Double Ratchet/MLS are complete | JNI path is AES-GCM session wrapper and MLS still uses placeholder crypto/sign logic | Miscommunicates user security posture and invites unsupported expectations |
| `TEST_RESULTS.md` says “READY FOR BUILD & DEPLOYMENT” | `TEST_HARNESS_SCOPE.md` makes it explicit tests are structural wiring checks | Inflates confidence in deployment readiness |
| Desktop settings describe mining, bridge, covert channel, and device-trust that rely on TODO code | `settings_cryptogram.cpp` marks those paths unsupported or short of implementation | UI-level feature inflation causes false readiness claims |
| Feature docs reference modules excluded from `Telegram/CMakeLists.txt` | `DESKTOP_BUILD_ALIGNMENT.md` shows source-only files with no build entry | Readers assume runtime support they cannot get |

## Sequenced Implementation Phases
1. **Phase 1 – Truth Baseline (Week 0-1)**
   - Owner: Docs (primary) with Android, Desktop, QA reviewing.
   - Actions: normalize vocabulary across root/docs/UI; remove/flag “production/complete” language; add capability-state table per platform.
   - Outcome: no contradictory signals across documentation surfaces; canonical status page published.
2. **Phase 2 – Surface Gating (Week 1-2, parallel)**
   - Owners: Android + Desktop engineering with Docs.
   - Actions: relabel or hide UI toggles/status text for unvalidated features; mark experimental paths; align `CryptogramNative.getFeatureStatus()` with real capability checks.
   - Outcome: UI claims do not outpace proven runtime behavior.
3. **Phase 3 – Android Crypto Refinement & Verification (Week 2-6)**
   - Owner: Android (primary), with QA + Build/Infra for tests.
   - Outcome: Runtime fully matches product/docs claims; storage is secured and hardened.
4. **Phase 4 – Desktop Feature Integrity (Week 2-6, parallel)**
   - Owner: Desktop engineering with QA + Docs.
   - Actions: implement missing runtime backends or gate settings behind experimental/disabled states; resolve `Telegram/CMakeLists.txt` wiring vs docs; add CI jobs that compile/run the feature tests beyond static grep.
   - Outcome: every visible desktop toggle has verified effect or is explicitly disabled; build matrix published.
5. **Phase 5 – Verification Hardening & Release Gate (Week 6-8)**
   - Owner: QA + Build/Infra, coordinating with Android/Desktop/Docs.
   - Actions: split validation tiers (static/build/runtime/crypto/device), enforce release gate (passes + docs sign-off) before any “ready” language; publish signed status snapshot.
   - Outcome: “ready/production” claims only appear after gate pass; roadmap closes with auditable evidence.

## Critical Path & Priorities
- Phase 1 documentary baseline -> Phase 2 surface gating -> Phase 3 Android crypto correctness -> Phase 5 verification gate. Desktop Phase 4 can progress in parallel with Phase 3 but must finish before final release language.
- High-impact fixes before others: (a) docs/status normalization, (b) UI/feature gating for false-positive toggles, (c) Android runtime cryptography correctness, (d) QA verification gate.
- Supporting priorities: publish desktop build matrix, run concrete runtime tests, remove placeholder MLS crypto, and document failure/down-grade behavior.
