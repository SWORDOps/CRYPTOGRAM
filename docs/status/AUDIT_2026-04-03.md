# CRYPTOGRAM Audit - 2026-04-03

This audit compares the current repository state against the active documentation surface and identifies the highest-value gaps to patch next.

## Scope

- Root docs and status pages
- Android CRYPTOGRAM implementation under `TMessagesProj/`
- Desktop CRYPTOGRAM implementation under `Telegram/`
- In-repo automated test surface under `tests/unit/`

## Executive Summary

The repo contains real CRYPTOGRAM code on both Android and desktop, but the strongest gaps are not missing files. They are mismatches between what some docs and UI status surfaces imply and what the code actually proves.

The highest-priority gaps are:

1. The Android "Double Ratchet" JNI wrapper is not a full Double Ratchet implementation.
2. MLS still contains placeholder cryptographic operations.
3. Android docs/status pages overstate readiness and test completeness.
4. Desktop settings expose several TODO-backed features as if they are available.
5. Some older audit notes are already narrower than the current code after recent self-test and build-wiring changes.

## Findings

### 1. Android feature-status reporting is better aligned after recent self-test changes

Severity: informational

Evidence:
- `CryptogramNative.getFeatureStatus()` now delegates Double Ratchet and MLS checks to native methods: [CryptogramNative.kt](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/src/main/java/org/telegram/messenger/cryptogram/CryptogramNative.kt#L58)
- JNI exports `nativeCheckDoubleRatchet()` and `nativeCheckMLS()`: [CryptogramWrapper.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/CryptogramWrapper.cpp#L597)

Read:
- This is an improvement over pure helper-based status reporting.
- It is still a local self-check, not a full interoperability or external security validation story.

### 2. The Android Double Ratchet wrapper is a simpler AES-GCM session wrapper

Severity: high

Evidence:
- The JNI state JSON identifies the active protocol as `"AES-256-GCM session"` rather than Double Ratchet: [CryptogramWrapper.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/CryptogramWrapper.cpp#L181)
- Session init creates a per-user session with one key and counters, not a ratchet state machine: [CryptogramWrapper.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/CryptogramWrapper.cpp#L197)
- Encrypt/decrypt use `encryptAesGcm` / `decryptAesGcm` directly with counters: [CryptogramWrapper.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/CryptogramWrapper.cpp#L219), [CryptogramWrapper.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/CryptogramWrapper.cpp#L265)
- The Java helper and settings present this as Signal/Double Ratchet to users: [CryptogramMessageHelper.java](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/src/main/java/org/telegram/messenger/cryptogram/CryptogramMessageHelper.java#L61), [CryptogramSettingsActivity.java](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/src/main/java/org/telegram/ui/CryptogramSettingsActivity.java#L246)

Patch target:
- Either wire the wrapper to the real signal-protocol implementation in `data_signal_protocol.*`, or downgrade UI/docs language to "session encryption" until that is done.

### 3. MLS still uses placeholder cryptographic primitives

Severity: high

Evidence:
- X448 key generation is explicitly marked placeholder: [data_mls_protocol.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/data_mls_protocol.cpp#L878)
- Signing is a placeholder returning random bytes: [data_mls_protocol.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/data_mls_protocol.cpp#L892)
- Verification is a placeholder size check: [data_mls_protocol.cpp](/media/john/NVME_STORAGE10/CRYPTOGRAM/TMessagesProj/jni/cryptogram/data_mls_protocol.cpp#L899)

Patch target:
- Replace placeholder X448/sign/verify paths or mark the MLS ciphersuite surface as experimental in UI/docs.

### 5. Hardware-backed Android key storage is not evidenced by Android-side implementation

Severity: medium

- The Android tree does not show `AndroidKeyStore`, `KeyStore`, `KeyGenParameterSpec`, or `KeyProperties` usage in `TMessagesProj`: local search returned no matches

Patch target:
- Either add a real Android Keystore-backed key path, or narrow docs to "hardware-backed/TSM-oriented design hooks" instead of Android KeyStore capability.

## Patch Order

### Phase 1: truth-in-docs and truth-in-status

1. Fix `CryptogramNative.getFeatureStatus()` to use real capability checks.
2. Rewrite `docs/status/TEST_RESULTS.md` to remove deployment/completeness claims.
3. Mark MLS and Double Ratchet Android status as experimental/partial wherever user-facing text still overstates them.

### Phase 2: Android crypto correctness

1. Replace the AES-GCM session wrapper with real SignalProtocol ratchet integration, or rename the feature surface honestly.
2. Replace MLS placeholder X448/sign/verify code.
3. Add tests for feature-status reporting and message-flow behavior.

### Phase 3: desktop unfinished features

1. Disable or relabel Monero mining, covert channel, bridge, and device-trust settings until implemented.
2. Keep OpenVINO translation marked as experimental until on-device runtime inference is wired.
3. Add focused tests for any desktop modules kept user-visible.

## Recommended Immediate Patches

If patching starts now, the fastest high-value sequence is:

1. Fix Android feature-status false positives.
2. Rewrite `docs/status/TEST_RESULTS.md`.
3. Add explicit "experimental/partial" labels in Android and desktop settings text for unfinished crypto/features.
